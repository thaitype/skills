# claude-context: How Context Calculation Works

A Python script that reads Claude Code session files (`.jsonl`) and calculates context window usage to match the official `/context` command.

## Session file location

Claude Code stores conversation logs at:

```
~/.claude/projects/<project-dir>/<session-id>.jsonl
```

Each `.jsonl` file is one session. Each line is a JSON object representing a message, tool call, or system event.

## JSONL message structure

Lines with `type: "assistant"` contain usage data:

```json
{
  "type": "assistant",
  "message": {
    "role": "assistant",
    "model": "claude-opus-4-6",
    "usage": {
      "input_tokens": 1,
      "output_tokens": 515,
      "cache_read_input_tokens": 50144,
      "cache_creation_input_tokens": 949
    }
  }
}
```

The `usage` fields mean:
- `input_tokens` — uncached input tokens (newly sent, not in cache)
- `cache_read_input_tokens` — tokens read from Anthropic's prompt cache
- `cache_creation_input_tokens` — tokens written to cache for the first time
- `output_tokens` — tokens generated by the model

## How context used is calculated

The total context used for a single API call is:

```
context_used = input_tokens + cache_read_input_tokens + cache_creation_input_tokens
```

This sum represents **everything sent to the model** in one turn — the full conversation history, system prompt, tools, etc. Cached or not, it all counts toward the context window.

The script uses the **latest** assistant message's usage, not a sum across all messages. Each API call sends the entire context, so the most recent call reflects the current state.

### Example

```
input_tokens:                  1       (uncached)
cache_read_input_tokens:   50,144      (read from cache)
cache_creation_input_tokens:  949      (newly cached)
───────────────────────────────────
context_used:              51,094      (total sent to model)
```

For a 1M context window: `51,094 / 1,000,000 = 5.1%` usage.

## Why not sum all messages?

Each API call sends the **full conversation** to the model. The `input_tokens` in each response already includes all prior messages. Summing across messages would double/triple count everything.

Think of it like checking disk usage — you check the current total, not add up every write operation.

## Context window sizes

| Model | Context Window |
|-------|---------------|
| Claude Opus 4.x | 1,000,000 |
| Claude Sonnet 4.x | 1,000,000 |
| Claude Haiku 4.x | 1,000,000 |
| Claude 3.5 Sonnet/Haiku | 1,000,000 |
| Claude 2.1 | 200,000 |
| Claude 2.0 | 100,000 |

## Auto-compact buffer

Claude Code reserves 3.3% of the context window as a buffer for auto-compaction:

```
buffer = context_window * 0.033
```

For a 1M window: `1,000,000 * 0.033 = 33,000 tokens`

When context usage approaches `context_window - buffer`, Claude auto-compacts the conversation by summarizing older messages to free up space.

## After compaction

When auto-compact or `/compact` runs:
1. Old messages are summarized into a single compact message
2. This message is injected as a `type: "user"` with text starting with: `"This session is being continued from a previous conversation"`
3. There is **no new assistant message with usage data** immediately after compaction

Since no usage data exists post-compact, the script estimates context from:

```
estimated = 18,000 (system overhead) + compact_summary_chars / 4 + post_compact_message_chars / 4
```

The `~4 chars/token` ratio is a rough approximation. Once the user sends a new message and gets a response, the script picks up real usage data again.

## Comparison with official `/context`

| What | Official `/context` | This script |
|------|-------------------|-------------|
| Source | Live internal state | JSONL file parsing |
| Context used | Exact | Exact (from usage data) |
| Post-compact | Exact | Estimated (~4 chars/token) |
| System breakdown | Yes (prompt, tools, memory, skills) | No (total only) |
| Autocompact buffer | 3.3% of window | 3.3% of window |

The script matches the official report when usage data is available. After compaction with no new API call, there can be a small estimation gap until the next turn.
